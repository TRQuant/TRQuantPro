## 📚 核心概念

### 版本号规则

- **MAJOR**：主版本号，不兼容的API修改
- **MINOR**：次版本号，向下兼容的功能新增
- **PATCH**：修订版本号，向下兼容的问题修正

### 更新类型

- **功能更新**：新增功能、功能增强
- **Bug修复**：问题修复、错误修正
- **性能优化**：性能提升、资源优化
- **文档更新**：文档完善、示例更新

<h2 id="section-a-2-1">📌 A.2.1 版本说明</h2>

版本号遵循语义化版本规则，文档版本与TRQuant版本对应。

### 版本号规则

#### 语义化版本

版本号格式：`MAJOR.MINOR.PATCH`

- **MAJOR**：主版本号，不兼容的API修改
- **MINOR**：次版本号，向下兼容的功能新增
- **PATCH**：修订版本号，向下兼容的问题修正

**示例**：
- `1.0.0`：初始版本
- `1.1.0`：新增功能（向下兼容）
- `1.1.1`：Bug修复（向下兼容）
- `2.0.0`：重大更新（不兼容）

### 版本类型

- **稳定版**：经过充分测试的稳定版本
- **开发版**：正在开发中的版本
- **预览版**：功能预览版本

### 更新频率

- **主版本**：重大更新，不频繁
- **次版本**：功能更新，定期发布
- **修订版本**：Bug修复，及时发布

<h2 id="section-a-2-3">🔄 A.2.3 自动生成</h2>

更新日志支持从多个来源自动生成。

### 自动生成来源

#### evidence_server

开发历程记录，包括：
- 功能开发记录
- 问题解决记录
- 设计决策记录

#### git

版本标签和提交历史，包括：
- 版本标签（git tag）
- 提交信息（git commit）
- 代码变更（git diff）

#### report_server

变更报告，包括：
- 功能变更报告
- Bug修复报告
- 性能优化报告

### 自动生成流程

```
1. 从evidence_server获取开发历程
   ↓
2. 从git获取版本标签和提交历史
   ↓
3. 从report_server获取变更报告
   ↓
4. 合并生成更新日志
   ↓
5. 更新文档
```

### 手动更新

如果自动生成失败，可以手动更新：

1. 查看git提交历史
2. 查看issue和PR
3. 整理功能更新和Bug修复
4. 更新更新日志文档
